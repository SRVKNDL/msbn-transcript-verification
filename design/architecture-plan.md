## 1. High-Level Flow

This describes the path from a transcript PDF landing in S3 to a reviewer seeing a flagged
application in the dashboard. There are two logical halves: the **processing pipeline**
(automated) and the **review workflow** (human-in-the-loop).

### 1.1 Ingestion

1. An MSBN staff member uploads one or more PDF documents for an application (transcript,
   diploma, CEA report, affidavit of graduation) to the raw S3 prefix via a pre-signed URL
   generated by the intake API.
2. The S3 upload fires an event notification that invokes the **Intake Lambda**.
3. Intake Lambda creates an application record in DynamoDB with status `INTAKE_COMPLETE`, records the
   S3 path and document type metadata, and starts a **Step Functions Standard Workflow**
   execution, passing the application ID and document manifest.

### 1.2 Processing Pipeline (Step Functions)

The workflow executes the following states in order:

```
[Update Status: EXTRACTING]
       |
       v
[Parallel: Extract each document]
  |-- ExtractLambda (document 1 — transcript)
  |-- ExtractLambda (document 2 — diploma)
  `-- ExtractLambda (document N — ...)
       |
       v
[AggregationLambda — cross-document field comparison (CROSS_* fields)]
       |
       v
[Update Status: EVALUATING]
       |
       v
[RuleEngineLambda — single-document rules (PHYS, CONT, PROG per doc)]
       |
       v
[CrossDocLambda — cross-document rules (CROSS_001-003)]
       |
       v
[PopulationCheckLambda — population-level rules (POP_001-003)]
       |
       v
[Update Status: READY_FOR_REVIEW]
       |
       v
[NotifyLambda — email reviewer queue via SNS]
```

**Standard Workflow** (not Express) is used because Standard persists full execution history,
which is part of the audit trail. See Section 7, SP-9.

**Parallel extraction** uses a Step Functions `Map` state over the document manifest so all
documents for one application are extracted concurrently. The AggregationLambda step runs
after the `Map` state completes, once all per-document extractions are available.

### 1.3 Extraction Detail (ExtractLambda)

ExtractLambda handles single-page extraction against one document at a time. It does not
compare fields across documents; that comparison runs in the AggregationLambda after all
per-document extractions are complete (see Section 1.4).

For each document:

1. Retrieve the PDF from S3.
2. Convert each page to a PNG image using a PDF rendering library (Lambda container image).
3. Write page images to `processed/{application_id}/page_{doc_type}_{n}.png` in S3.
4. Call **Bedrock Nova** with each page image plus a structured extraction prompt. The prompt
   asks for facts only — no analysis (see Section 3).
5. Aggregate per-page responses into a single `extraction.json` document.
6. Write `extraction.json` to `processed/{application_id}/extraction_{doc_type}.json` in S3.
7. Write the S3 path to DynamoDB; mark document status `EXTRACTED`.

### 1.4 Aggregation Detail (AggregationLambda)

Runs once per application after all per-document extractions complete. Produces the
cross-document fields defined in `design/extraction-vocabulary.md` Section 4
(`signature_match_across_documents`, `institution_name_match`, `applicant_name_match`,
`dates_match_across_documents`).

1. Read all `extraction_{doc_type}.json` files for the application from S3.
2. Compare the relevant fields across documents to populate the Section 4 vocabulary values.
3. Write `processed/{application_id}/aggregation.json` to S3.
4. Write the S3 path to DynamoDB.

CrossDocLambda consumes `aggregation.json` rather than re-reading individual extraction files.

### 1.5 Rule Evaluation Detail (RuleEngineLambda)

1. Read `extraction.json` for the target document from S3.
2. Load reference lookup tables from S3 (`reference/` prefix): grading scales, accreditation
   list, known nursing course names, institution seal types, diploma mill phrase list.
3. Apply every applicable Python rule from `docs/requirements-draft.md` Section 6. Rules are
   pure functions: `rule_fn(extraction: dict, reference_tables: dict) -> list[Flag]`.
4. Write each generated `Flag` as a DynamoDB item (see Section 4).

### 1.6 Human Review (Dashboard)

1. A reviewer authenticates via Cognito and opens the dashboard.
2. The dashboard calls the **Dashboard API Lambda** via API Gateway to fetch the application
   record, flag list, and presigned URLs for the page images stored in S3.
3. For each page, the dashboard renders the page image and overlays flag highlights using the
   bounding box or text-span coordinates stored with each flag.
4. The reviewer reads the rationale for each flag, views the highlighted source location, and
   marks each flag as **Confirmed** or **Overridden**, optionally adding notes.
5. When all flags are resolved, the reviewer submits a final decision. The Dashboard Lambda
   writes the decision and all reviewer actions to the DynamoDB audit trail and updates
   application status to `DECIDED`.
6. If MSBN requires Nursys/NPDB reporting (denial case), the Step Functions workflow triggers
   a **ReportingCheckpoint** state that surfaces a mandatory acknowledgment field in the
   dashboard before the application can be closed (NRB Policy 7).

## 2. AWS Services and Justification

### 2.1 Core Pipeline Services

| Service | Role | Hub Guide Alignment |
|---|---|---|
| **Amazon S3** | Raw PDFs, page images, extraction JSON, reference tables, frontend assets | Explicitly preferred over EBS for data storage. Essential for event-driven intake. |
| **AWS Lambda** | All compute: intake, extract, aggregation, rule engine, cross-doc, population check, notify, dashboard API | Primary serverless compute. Hub Guide: "Use AWS Lambda for API endpoints and orchestration logic." Eight distinct Lambda functions, each with single responsibility. |
| **AWS Step Functions (Standard)** | Pipeline orchestration, error handling, retry logic, audit history | Hub Guide: "Use AWS Step Functions for workflow orchestration." Standard (not Express) required because audit history is a project hard constraint. |
| **Amazon Bedrock (Nova Lite / Pro)** | PDF content extraction, visual document analysis | Hub Guide: "Prefer Amazon Nova Models in Bedrock." Nova Lite for routine extraction; Nova Pro reserved for complex reasoning (ambiguous course names, seal type classification, diploma mill pattern detection). |
| **Amazon DynamoDB** | Application records, flags, audit trail, reviewer decisions | Hub Guide: "Use Amazon DynamoDB for NoSQL data storage needs." On-demand billing mode; no capacity planning required for POC volume. |
| **Amazon API Gateway (HTTP API)** | Dashboard REST backend | Hub Guide recommends HTTP API (cheaper than REST API) unless REST-specific features are needed. For this POC, HTTP API is sufficient. |
| **Amazon Cognito** | Reviewer authentication and JWT issuance | Serverless, no EC2. Free for ≤ 50,000 MAU. Required because the dashboard exposes PII-adjacent data. Not in the Hub Guide's "core five" but is the correct serverless auth primitive — adding this requires no justification under the "no EC2/no SageMaker" rule. |
| **Amazon CloudWatch** | Structured logs, Lambda metrics, budget alarm integration | Free tier covers POC volume. Hub Guide: "Enable CloudWatch Logs for all Lambda functions." |
| **Amazon SNS** | Email notification to reviewer queue when application is ready | Negligible cost. Avoids building a notification system inside Lambda. |
| **Amazon CloudFront** | HTTPS CDN for React dashboard assets hosted on S3 | Free tier covers POC traffic volume. Required for proper HTTPS on the S3-hosted SPA without a load balancer. |

### 2.2 Services Outside the Core Five

Two services beyond Lambda/Bedrock/S3/DynamoDB/API Gateway/Step Functions are used.
Both are serverless and both have explicit cost justifications.

**Amazon Cognito**
Justification: The dashboard will be accessible over the internet and will display
application status, flag details, and applicant metadata. Without authentication, any user
with the URL could access this data. Cognito User Pools provide JWT-based auth with
MFA support, integrate directly with API Gateway's JWT authorizer, and cost nothing
at POC user counts. The alternative — API keys embedded in the frontend — is insecure
and not auditable.

**Amazon Textract** *(Phase 3 option, not required for Phase 2)*
Justification: The explainability requirement specifies "page + bounding box or text span"
per flag. Bedrock Nova multimodal can return page numbers and text spans, which are
sufficient for Phase 2. However, precise pixel-level bounding boxes (needed for tight
highlight overlays in the dashboard) require Textract, which returns exact normalized
coordinates per word block. Cost at POC volume: $1.50 per 1,000 pages — under $2/month
for 100 transcripts. Recommend deferring Textract to Phase 3 after confirming whether
Nova's text spans are sufficiently precise for MSBN reviewers.

### 2.3 Services Explicitly Rejected

| Service | Why Rejected |
|---|---|
| EC2 | Hub Guide and CLAUDE.md: no EC2. All compute fits in Lambda; the only size concern (PDF rendering) is solved with Lambda container images. |
| Amazon SageMaker | Hub Guide and CLAUDE.md: no SageMaker unless justified. No custom model training is required; Bedrock Nova handles all AI inference. |
| RDS / Aurora | DynamoDB is sufficient for the access patterns (key-value + sparse queries). RDS charges even when idle, violating cost rules. |
| OpenSearch Serverless | Not needed: no full-text search requirement in the pipeline. Rule evaluation is against structured extracted data, not free-text retrieval. (Re-evaluate if Bedrock Knowledge Base is added for the accreditation reference lookup.) |
| Elastic Load Balancer | CloudFront handles HTTPS for the static frontend; API Gateway handles routing. No ELB required. |

## 3. LLM vs. Deterministic Work Split

### The Boundary Rule

> Bedrock extracts **facts** from the document.
> Python rules evaluate those facts against **thresholds and reference data** to produce flags.
> Bedrock never produces a flag. Python never reads a raw PDF.

This boundary is a hard architectural constraint (CLAUDE.md: "Extraction via LLM. Rules via
plain Python, NOT prompts"). It ensures every flag is traceable to deterministic logic,
satisfying the no-black-box explainability requirement.

### 3.1 What Bedrock Does

Bedrock Nova receives page images and returns structured JSON. Every prompt is phrased as an
extraction question, never an evaluation question.

**Textual extraction (all document types):**
- Student name, date of birth (if present)
- Institution name, address, and phone number
- Document type and issue date
- Enrollment start date and graduation/conferral date
- Declared program name, degree title, major
- List of courses: name, credit hours or clock hours, grade value
- Reported cumulative GPA or overall average
- Graduation/conferral statement (exact text or null if absent)
- Language(s) visible in document
- Any text consistent with diploma mill promotion (exact excerpt)

**Visual extraction (multimodal, per page image):**
- Location description and quality assessment of the institution seal or logo
  (e.g., "embossed seal, center-bottom, clear edges" vs. "low-resolution stamp, pixelated")
- Print technology characterization (typewriter, dot matrix, laser, digital)
- Presence and location of signature blocks
- Any visually misaligned text blocks or inserted elements
- Presence of JPEG compression artifacts or scan banding

**Output contract:** Bedrock returns a structured JSON object conforming to a versioned schema
defined in the codebase. Every field includes a `source_page` integer and a `text_span`
string (the exact text excerpt that evidence came from). Bounding boxes are populated from
Textract if available, otherwise null.

**What Bedrock is never asked:**
- Whether a document is suspicious
- Whether a course name is legitimate
- Whether a GPA is consistent
- Whether any flag should be raised
- Any question whose answer implies a fraud determination

### 3.2 What Python Rules Do

The RuleEngineLambda receives the extraction JSON and reference tables and runs every
applicable rule as a pure Python function. Examples of the logic (not production code,
illustrative only):

```
CONT_005:
  computed = mean(extraction["course_grades"])
  reported = extraction["reported_gpa"]
  delta = abs(computed - reported) / reported
  if delta > THRESHOLD:
    flag(rule="CONT_005", severity="High",
         rationale=f"Reported GPA {reported} differs from computed {computed:.2f} by {delta:.1%}.",
         source_location=extraction["reported_gpa_location"])

CONT_002:
  age_at_enrollment = enrollment_year - applicant_dob_year
  if age_at_enrollment < 16:
    flag(rule="CONT_002", ...)
  if graduation_date < enrollment_date:
    flag(rule="CONT_002", ...)

PHYS_001:
  if extraction["seal_quality"] in ("pixelated", "low_resolution", "copy_artifact"):
    flag(rule="PHYS_001", ...)

PROG_001:
  normalized = normalize_institution_name(extraction["institution_name"])
  if normalized not in reference_tables["accreditation_lookup"]:
    flag(rule="PROG_001", ...)
  if any(phrase in extraction["diploma_mill_text"] for phrase in reference_tables["diploma_mill_phrases"]):
    flag(rule="PROG_001", ...)
```

All rule thresholds (e.g., `> 10%` GPA delta, `> 90 days` date mismatch, `< 16` years old
at enrollment) are defined in a single `rules_config.yaml` file, not hardcoded. MSBN can
adjust thresholds without touching Python code.

### 3.3 Reference Tables

Reference tables are JSON files in `s3://{bucket}/reference/`. They are loaded by
RuleEngineLambda at startup and cached in Lambda memory across warm invocations. Tables:

| File | Used by |
|---|---|
| `accreditation_lookup.json` | PROG_001 |
| `grade_scale_by_country.json` | CONT_001 |
| `country_official_languages.json` | CONT_004 |
| `known_nursing_courses.json` | CONT_003 |
| `institution_seal_types.json` | PHYS_002 (embossed vs. stamped) |
| `diploma_mill_phrases.json` | PROG_001 |
| `cea_approved_agencies.json` | Section 3 / SP-5 |

Tables are seeded from the requirements draft; MSBN provides official values before production.

## 4. Data Model

### 4.1 S3 Prefix Structure

```
s3://msbn-transcripts-{env}/
│
├── raw/
│   └── {application_id}/
│       └── {doc_type}_{original_filename}.pdf    # immutable, never overwritten
│
├── processed/
│   └── {application_id}/
│       ├── extraction_TRANSCRIPT.json            # Nova structured output
│       ├── extraction_DIPLOMA.json
│       ├── aggregation.json                      # cross-document fields (AggregationLambda output)
│       ├── page_TRANSCRIPT_1.png                 # per-page images for dashboard
│       ├── page_TRANSCRIPT_2.png
│       └── textract_TRANSCRIPT.json              # raw Textract response (Phase 3)
│
├── reference/
│   ├── accreditation_lookup.json
│   ├── grade_scale_by_country.json
│   ├── country_official_languages.json
│   ├── known_nursing_courses.json
│   ├── institution_seal_types.json
│   ├── diploma_mill_phrases.json
│   └── cea_approved_agencies.json
│
└── frontend/
    └── (React build artifacts — index.html, JS bundles, etc.)
```

Bucket name: `msbn-transcripts-{env}` where env is `dev` or `prod`.
Versioning is enabled on `raw/` only. Encryption: SSE-S3 on all objects.
Public access: blocked. Dashboard accesses page images via Lambda-generated presigned URLs
(5-minute expiry).

### 4.2 DynamoDB — Single Table: `msbn-applications`

**Table design: single-table with record-type discrimination via sort key prefix.**

| Attribute | Type | Description |
|---|---|---|
| `PK` | String (partition key) | `APP#{application_id}` |
| `SK` | String (sort key) | Record type + qualifier (see below) |
| `entity_type` | String | Mirrors SK prefix: `METADATA`, `DOCUMENT`, `FLAG`, `AUDIT`, `DECISION` |
| `GSI1PK` | String | Used for queue queries (see GSIs) |
| `GSI1SK` | String | Used for queue queries |
| `GSI2PK` | String | Used for POP_001 (license number dedup) |
| `TTL` | Number | Unix timestamp; set only on ephemeral items if needed |

**Sort key patterns:**

| SK Pattern | Record type | Purpose |
|---|---|---|
| `METADATA` | Application metadata | Status, timestamps, reviewer assignment, Step Functions ARN |
| `DOCUMENT#{doc_type}` | Per-document state | S3 raw path, S3 extraction path, extraction status, model version used |
| `FLAG#{rule_code}#{seq}` | Individual flag | See flag schema below |
| `AUDIT#{ISO8601_timestamp}` | Audit event | Immutable log of every state change and reviewer action |
| `DECISION` | Final decision | Reviewer ID, decision (APPROVED/DENIED/DEFERRED), timestamp, notes |

**METADATA item attributes (key ones):**

```
PK:                  APP#a7f3b...
SK:                  METADATA
status:              INTAKE_COMPLETE | EXTRACTING | EVALUATING | READY_FOR_REVIEW | DECIDED
applicant_name:      extracted after first document; null until extracted
submission_ts:       ISO8601 timestamp
last_updated_ts:     ISO8601 timestamp
assigned_reviewer:   Cognito sub or null
workflow_arn:        Step Functions execution ARN (links to full execution history)
document_count:      integer
flag_count:          integer (maintained by rule engine, incremented atomically)
high_severity_count: integer
```

**FLAG item attributes:**

```
PK:                  APP#a7f3b...
SK:                  FLAG#CONT_005#001
rule_code:           CONT_005
severity:            High | Medium | Low
rationale:           "Reported GPA 3.4 differs from computed mean 2.74 by 23.9%."
source_location:     {see Section 5}
reviewer_status:     OPEN | CONFIRMED | OVERRIDDEN
reviewer_id:         Cognito sub or null
reviewer_ts:         ISO8601 timestamp or null
reviewer_notes:      string or null
extraction_s3_ref:   s3://...processed/.../extraction_TRANSCRIPT.json
```

**AUDIT item attributes:**

```
PK:                  APP#a7f3b...
SK:                  AUDIT#2026-04-12T14:32:01Z
actor:               system | {cognito_sub}
event_type:          STATUS_CHANGED | FLAG_RAISED | FLAG_CONFIRMED | FLAG_OVERRIDDEN |
                     REVIEWER_ASSIGNED | DECISION_SUBMITTED | NURSYS_ACKNOWLEDGED
previous_state:      JSON snapshot of relevant prior state
new_state:           JSON snapshot of new state
```

Audit items are append-only. No audit item is ever updated or deleted.

**Global Secondary Indexes:**

| GSI | GSI PK | GSI SK | Query purpose |
|---|---|---|---|
| `GSI1-ReviewQueue` | `status` | `submission_ts` | Dashboard: list all applications in READY_FOR_REVIEW order |
| `GSI2-LicenseDedup` | `license_number#country` | `PK` | POP_001: find collisions on same license number |
| `GSI3-InstitutionCluster` | `institution#program#grad_year` | `PK` | POP_002/003: cluster applications by school |

GSI2 and GSI3 are populated only on the METADATA item after extraction. GSIs add cost only
when queried; at POC volume this is negligible.

### 4.3 Audit Trail Completeness

The audit trail must answer: "Who did what, when, and what did the system know at the time?"

Every state transition writes an AUDIT item. This includes:
- Pipeline step starts and completions (actor: `system`)
- Model ID and prompt version used for each extraction (captured in DOCUMENT item)
- Every flag raised (actor: `system`, event: `FLAG_RAISED`)
- Every reviewer action on a flag (actor: `{cognito_sub}`, event: `FLAG_CONFIRMED` or `FLAG_OVERRIDDEN`)
- Final decision submission (actor: `{cognito_sub}`, event: `DECISION_SUBMITTED`)
- Nursys acknowledgment for denial cases (actor: `{cognito_sub}`, event: `NURSYS_ACKNOWLEDGED`)

The Step Functions execution ARN stored in METADATA lets anyone reconstruct the full
machine-level execution history independent of the DynamoDB audit trail, providing
redundant auditability.

## 5. Explainability: Flag-to-Source Traceability

### 5.1 The Requirement

Every flag must include "rule code, human-readable rationale, and source location in the
transcript (page + bounding box or text span)" (CLAUDE.md architectural principles).

### 5.2 Data Contract for `source_location`

Every `FLAG` item's `source_location` attribute is a JSON object conforming to this schema:

```json
{
  "document_type": "TRANSCRIPT",
  "page_number": 2,
  "text_span": {
    "matched_text": "Overall GPA: 3.4",
    "context_before": "...Adult Med/Surg 2.2 | Pharmacology 2.6 | Psych 2.9 | ",
    "context_after": " | Enrollment: 01/07/2008"
  },
  "bounding_box": {
    "left": 0.42,
    "top": 0.78,
    "width": 0.15,
    "height": 0.03,
    "coordinate_system": "normalized_0_to_1"
  },
  "extraction_field_path": "reported_gpa",
  "extraction_s3_key": "processed/APP#a7f3b/extraction_TRANSCRIPT.json"
}
```

`text_span` is always populated (sourced from Bedrock's extraction output).
`bounding_box` is populated when Textract is in the pipeline; otherwise null. The dashboard
falls back to text-span highlighting when bounding boxes are absent.

For PHYS rules (visual anomalies), `text_span` is replaced by a `visual_region` descriptor:

```json
{
  "document_type": "TRANSCRIPT",
  "page_number": 1,
  "visual_region": {
    "description": "Institution seal, bottom-center of page",
    "detail": "Seal appears stamped; institution reference table specifies embossed seal.",
    "bounding_box": { ... }
  },
  "extraction_field_path": "seal_type"
}
```

### 5.3 Dashboard Rendering Flow

1. Dashboard requests flag list for application → API Lambda returns flags with `source_location`.
2. Dashboard requests page image → API Lambda returns presigned S3 URL for
   `processed/{app_id}/page_{doc_type}_{n}.png` (5-minute expiry).
3. Browser renders page image at full resolution in an `<img>` tag.
4. For each flag on that page, browser overlays a semi-transparent highlight `<div>` using
   `bounding_box` coordinates scaled to the rendered image dimensions.
   If bounding box is null, the dashboard highlights the `matched_text` via text search
   in the extracted text sidebar instead.
5. Clicking a flag highlight opens a side panel showing: rule code, severity, rationale,
   the matched text or visual region description, and the confirmation/override controls.

### 5.4 Extraction Prompt Engineering for Traceability

The Bedrock extraction prompt explicitly requests source references. Every structured field
in the extraction schema is paired with a `_page` and `_span` sibling:

```
"reported_gpa": 3.4,
"reported_gpa_page": 2,
"reported_gpa_span": "Overall GPA: 3.4"
```

This is enforced via the extraction schema (versioned in the codebase), not left to Nova's
discretion. Prompt version is recorded in the DOCUMENT DynamoDB item so the extraction can
be reproduced if the prompt changes.

- 100 applications/month, each with 2 documents (transcript + diploma) averaging 5 pages

- All services in us-east-1

| **S3 requests** | ~20,000 GET/PUT | $0.0004/1,000 PUT | ~$0.01 |
| **API Gateway (HTTP)** | ~5,000 requests | $1.00/million | ~$0.01 |

**Development and testing overhead:** Expect 10–20× the production pipeline runs during
Phase 2–3 development (repeated test runs, prompt iteration). At 10× multiplier, development
spend peaks at ~$5–20/month during active sprints. Still well within the $1,000 budget.

**Budget risk flags:**

| Risk | Trigger | Mitigation |
|---|---|---|
| Bedrock token explosion | Long system prompts or very large transcripts (20+ pages) | Set max-token limits in extraction Lambda; test with actual transcript sizes before Phase 3 |
| Nova Pro overuse | If rule engine defaults to Pro for all calls instead of Lite | Enforce model selection in config; use Pro only for flagged visual reasoning steps |
| DynamoDB GSI hot reads | POP_001/002/003 queries run on every submission | GSIs use on-demand; add DynamoDB DAX caching only if query latency becomes a reviewer complaint |
| Textract at scale | If volume grows to 10,000 applications | At 50K pages, Textract = $75/month — still manageable, but monitor |
| Step Functions Standard history retention | AWS charges for executions stored > 90 days | Set execution history TTL; archive to S3 after 90 days via EventBridge rule |

The $1,000 budget is not at risk from any realistic POC workload. The alarms at $500/$750/$900
recommended by the Hub Guide should be configured in AWS Budgets. Given the estimates above,
hitting $500 would require roughly 25,000 transcript equivalents or a significant runaway
invocation loop.

## 7. Safe Practice Categories → System Components

Each of the 10 Safe Practices maps to either automated pipeline logic, reviewer workflow UI,
or both.

| Safe Practice | Type | System Component(s) |
|---|---|---|
| **SP-1** — Verification of Applicant Identity | Automated rule | CROSS_001 runs in CrossDocLambda after all documents extracted. Fuzzy name match across all document extractions. |
| **SP-2** — No Previous Denial of Licensure | Automated rule + workflow | POP_001 runs in PopulationCheckLambda (DynamoDB GSI2 query). Dashboard surfaces a "Nursys verification required" badge; reviewer must check Nursys and log acknowledgment before approving. |
| **SP-3** — No Disciplinary Action or BON Alert | Workflow only | No automated rule. Dashboard renders a mandatory "Nursys disciplinary check completed" checkbox per NRB Policy 6. This checkbox must be checked before the DECISION record can be submitted. Enforced in the Dashboard API Lambda (returns error if checkbox is not checked on submit). |

## 8. Open Questions Before Implementation Starts

These must be resolved before writing code. Grouped by which phase they block.

### Phase 2 Blockers (environment setup, data ingestion)

**Q1. PDF rendering in Lambda — container image or layer?**
The `ExtractLambda` needs a PDF-to-image library (poppler + pdf2image). Lambda zip
deployments cap at 250MB unzipped; poppler binaries alone are ~80MB. Options:
(a) Lambda container image (up to 10GB, simplest), (b) custom Lambda layer with
pre-compiled ARM64 poppler binaries. Decision needed before ExtractLambda is built.
Recommendation: container image for ExtractLambda only; all other Lambdas use standard zip.

**Q2. Textract now or Phase 3?**
Decide whether to include Textract in the Phase 2 extraction pipeline for precise bounding
boxes, or launch with Nova text spans and add Textract in Phase 3 after reviewing with
MSBN whether the highlight precision is acceptable. Deferring Textract simplifies Phase 2.

**Q3. Multi-document upload UX.**
Does MSBN upload all documents for one application in a single batch, or does the intake
form require one document per upload flow? This affects how the Step Functions manifest
is constructed and whether the intake Lambda waits for a complete set before triggering
the workflow.

**Q4. Reference table ownership and seeding.**
Who populates and updates the S3 reference tables (accreditation list, seal types, etc.)?
For Phase 2, the team seeds them from requirements-draft.md placeholders. Before Phase 4
(human review workflow), MSBN must supply official values or the HIGH-severity rules
cannot produce defensible flags.

### Phase 3 Blockers (rule engine + extraction)

**Q5. Bedrock multimodal page sending strategy.**
Should each page be sent to Bedrock as a separate invocation (simple, parallelizable,
consistent cost) or should multiple pages be batched per invocation using Nova's 300K
context window (fewer calls, lower per-token cost, harder to attribute source pages)?
Single-page invocations are recommended for traceability; revisit if cost is a concern.

**Q6. PHYS rule visual confidence thresholds.**
Bedrock Nova will describe seal quality and print technology in natural language. The Python
rule must map these descriptions to binary flag/no-flag decisions (e.g., "pixelated" → flag).
This requires a controlled vocabulary in the extraction prompt. The team needs to agree on
the vocabulary before building RuleEngineLambda.

**Q7. POP_002/003 cluster thresholds.**
POP_002 flags curricula with < X% overlap; POP_003 flags > 3× the 6-month baseline.
These thresholds cannot be validated until there is a corpus of synthetic test applications.
Confirm in Phase 5 (testing); ship POP rules as advisory-only Low severity until validated.

### Phase 4 Blockers (reviewer dashboard)

**Q8. Nursys integration scope.**
Does MSBN currently have a programmatic Nursys API integration, or is Nursys verification
always a manual browser step? If manual, the dashboard deep-link (Q2 in Section 7) is
the right approach. If MSBN has API credentials, a server-side Nursys lookup could be
added to the pipeline and cached in the METADATA item.

**Q9. Cognito user management.**
Who creates and manages reviewer accounts in the Cognito User Pool? For the POC, the team
can seed accounts manually. If MSBN has an existing identity provider (AD, Okta), Cognito
federation is possible but adds scope.

**Q10. CEA report expiration threshold.**
NRB Policy 11 requires an expiration date on CEA reports. The system can enforce this once
MSBN specifies the maximum acceptable report age (e.g., 2 years, 5 years). Until confirmed,
the system extracts the CEA report date and surfaces it in the dashboard without
auto-flagging.

### Cross-Phase

**Q11. Single-table vs. multi-table DynamoDB.**
The single-table design in Section 4 is correct for this access pattern. The risk is that
developers unfamiliar with single-table DynamoDB write inefficient queries. If the team
prefers, three separate tables (Applications, Flags, AuditTrail) are acceptable at POC
volume. Decide before Shushil begins the database schema implementation.

**Q12. Synthetic test corpus ownership.**
Section 7 of requirements-draft.md defines five MSBN-validated test cases (Cases A–E) as
synthetic data targets. Who builds the synthetic transcripts? Bishal (Data Engineer) is
the logical owner. These must exist before Phase 5 (testing) or the rule engine cannot
be validated. Recommend starting synthetic data generation in parallel with Phase 3.
